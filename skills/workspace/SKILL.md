Display workspace status and explain the multi-repo workspace conventions.

Arguments: $ARGUMENTS

## Workspace layout

Locate the workspace root by finding the nearest `.workspace` marker file in the current directory or ancestors.

**Discover categories** by reading `.claude/workspace-manager.local.md` at the workspace root. If the config file doesn't exist, fall back to listing subdirectories of `repos/`.

```
<workspace>/
├── repos/             # Read-only clones of all source repositories
│   └── <category>/    # One subdirectory per category (from config)
├── tasks/             # Active task worktrees (copy-on-write targets)
│   └── <task>/        # One directory per task — launch Claude Code here
│       ├── repos/     # Symlink to <workspace>/repos/ for convenient reading
│       └── <repo>/    # Git worktree for each modified repo
├── .claude/
│   └── workspace-manager.local.md  # Category configuration
├── .workspace         # Marker file identifying workspace root
└── CLAUDE.md          # Workspace rules (auto-generated by workspace-init)
```

## Launch pattern

Claude Code should be launched from a task directory, not the workspace root:
```
cd <workspace>/tasks/<task-name>
claude
```
This means the Bash sandbox automatically restricts writes to the task directory. The `repos/` symlink provides read access to all source repos. The PreToolUse hook separately blocks Edit/Write to repos/.

## Core rules

1. **repos/ is read-only.** Never edit files under repos/ directly. Enforced by two mechanisms: the Bash sandbox (blocks shell writes outside the task dir) and a PreToolUse hook (blocks Edit/Write to repos/).

2. **Copy-on-write via worktrees.** To modify a repo, create a git worktree in the task directory:
   ```bash
   git -C repos/<category>/<repo> worktree add \
     ./<repo> -b <task-name>/<branch-desc>
   ```
   Then edit files under `./<repo>/` instead.

3. **Reading is free.** Browse and read any file under repos/ without restriction. Use this to investigate issues, understand code, and gather context before making changes.

4. **Lazy worktree creation.** Don't create worktrees upfront. Only create them when you actually need to write to a repo. A task may read 10 repos but only write to 1-2.

5. **One task, one directory.** Each task gets a directory under tasks/. If a task spans multiple repos, each gets its own worktree under the same task directory.

6. **No automatic pulling.** Repos are not auto-updated. The user will ask to pull when they want to update.

## When invoked

1. Find the workspace root (.workspace marker)
2. Read `.claude/workspace-manager.local.md` to discover categories. If the file doesn't exist, detect categories by listing subdirectories of `repos/`.
3. List active tasks: `ls <workspace>/tasks/`
4. For each task, show which repos have worktrees
5. Show repo status per category: `ls <workspace>/repos/<category>/` for each discovered category
6. Present a summary of workspace state
